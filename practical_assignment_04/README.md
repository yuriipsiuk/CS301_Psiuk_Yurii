# Практична робота №4

Репозиторій містить розв'язання четвертої практичної роботи з CS301, виконаної Юрієм Псюком - практична група 4.
Для розв'язання була обрано БД для зберігання даних про футболістів, команди, суддів, стадіони та матчі.

---

## Структура репозиторію:

1. **`create_users.sql`** - запит, для генерування структури таблиць для бази даних
2. **`main.py`** - cкрипт на Python, який вставляє дані в таблиці; *(Написаний за допомогою штучного інтелекту GEMINI)*
3. **`requirements.txt`** - список необхідних бібліотек для генерації даних; *(Написаний за допомогою штучного інтелекту GEMINI)*
4. **`ERD.png`** - візуалізація діаграми сутність-зв'язок
5. **`indexes`** - містить необхідні індекси для пришвидшення запитів
6. **`create_function`** - запит, що створює funktion, яка рахує скільки голів забили футболісти команди за її індексом
7. **`create_procedure`** - запит, що створює процедуру, яка для заданого за id футболіста, при правильно введеній його поточній команді, переміщає його в нову
8. **`create_users`** - запит, що створює трьох користувачів з різним доступом
9. **`create_view`** - запит, що створює в'ю, яке вибирає усі матчі сьогоднішнього дня

---

## Структура БД:

* **table `teams`** - містить первинний ключ `team_id` і неключові дані про команди
* **table `footballers`** - містить первинний ключ, зовнішній ключ на `teams` (1:M відношення), один неключовий атрибут
* **table `footballer_info`** - містить первинний ключ, який є й зовнішнім до `footballers` (1:1 відношення), неключові атрибути, які відносяться до певного футболіста, але винесені в окрему таблицю, як приклад денормалізації
* **table `stadiums`** - містить первинний ключ і кілька неключових атрибутів стадіонів
* **table `referees`** -  містить первинний ключ і кілька неключових атрибутів рефері
* **table `tournaments`** - містить первинний ключ і кілька неключових атрибутів турніру
* **table `match_schedule`** - містить первинний ключ, і кілька атрибутів із `matches` як приклад денормалізації (1:1 відношення)
* **table `stadium_tournament`** - містить складений первинний ключ, є зв'язком між стадіонами і турнірами, де вони представлені, між ними відношення (M:M)
* **table `referee_tournament`** - містить складений первинний ключ, є зв'язком між рефері і турнірами, де вони представлені, між ними відношення (M:M)
* **table `matches`** - FACT TABLE, містить первинний ключ і зовнішні посилання на стадіон, рефері, турнір, часовий слот у `match_schedule`, а також посилання на дві команди, що беруть участь
