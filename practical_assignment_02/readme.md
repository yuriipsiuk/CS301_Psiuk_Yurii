Репозиторій містить розв'язання другої практичної роботи з CS301, виконаної Юрієм Псюком - практична група 4.

Репозиторій містить скрипт, де знаходяться 
1. Неоптимізований запит
2. Оптимізований запит разом з індексами для оптимізації виконання
3. Оптимізований запит з вимкнениими деякими характеристиками планувальника

Усі запити виконують задачу пошуку серед продуктів найпопулярнішої категорії з ідентифікатором > 9930, замовлених у 2022 році активними користувачами.
Запити беруть інформацію з трьох таблиць, згенерованих скриптом, який був у матеріалалх завдання.

Порівняльний аналіз через Explain Analyze
Конфігурація запиту        Execution Time      Тип сканування       Read          Запити до RAM
Неоптимізований            2131.486 мс         Seq Scan             360           4571944
Оптимізований(з індексами) 111.824 мс          Index Scan           0              80120
Оптимізований(без індексів) 611.132 мс         Seq Scan             0              17 378

Проблема неоптимізованого запиту полягала у об'єднуванні одних і тих самих таблиць у HAVING і WHERE і неможливістю використати індекси при фільтрації. 
Рішення що оптимізації:
1. Застосувати CTE для уникнення об'єднування одних і тих ж таблиць по кілька разів
2. Переписати умови фільтрації, щоб для них можна було застосувати індекси
3. Створити відповідні індекси на атрибути
Як результат, швидкість виконання зросла у 19 разів, вдалось уникнути ззапитів у диск і зменшити кількість запитів у RAM у 57 разів.

Демонстрація контролю планувальника
При унеможливлювані використання індексів і hashjoin запит почав втрачати у ефективності.
